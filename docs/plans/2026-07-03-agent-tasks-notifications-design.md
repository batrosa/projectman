# Дизайн: агент формирует задачи из документов + лента уведомлений + серверный мониторинг сроков

Дата: 2026-07-03. Статус: утверждён пользователем.

## Решения пользователя

1. Создание задач из документа — **с предпросмотром** (карточка в чате + кнопка «Создать N задач»), не автоматом.
2. Проверка «не взял в работу в течение часа» — **GitHub Actions ежечасно** дергает серверный монитор (Vercel Hobby cron умеет только раз в сутки).
3. Уведомление о просрочке — **раз в день, пока задача просрочена**.

## Контекст (что уже есть)

- Агент (`api/agent-chat`) читает `extractedText` файлов проекта; scoped по allowedProjects.
- Серверная отправка Telegram (`api/notify-telegram`), `telegramChatId` server-only.
- Клиентские напоминания в script.js (~5930-6070: «не взял в работу» с cooldown, «просрочено» день-после, «<20% времени») — работают только при открытой вкладке, шлют только исполнителю. **Отключаются** — сервер становится единственным источником (иначе двойные уведомления). Критерий «близится срок» меняется на «остался 1 день».
- Vercel Hobby: 12 функций максимум, сейчас 10. Cron на Hobby: максимум 2, только раз в сутки, время плавает в пределах часа.
- Репозиторий GitHub ПУБЛИЧНЫЙ — секреты только в GH Secrets / Vercel env, не в yaml/коде.

## 1. Данные

### Коллекция `agentNotifications/{id}` (новая)

```
{
  uid: string,              // получатель
  organizationId: string,
  taskId: string | null,
  projectId: string | null,
  type: 'overdue' | 'deadline_tomorrow' | 'not_taken_1h' | 'tasks_created',
  text: string,             // готовый текст по-русски
  createdAt: serverTimestamp,
  readAt: Timestamp | null
}
```

Правила: `read` — только свои (`request.auth.uid == resource.data.uid`); `update` — только свои и только поле `readAt`; `create`/`delete` — запрещены (пишет только сервер через Admin SDK).

Композитный индекс: `uid ASC, createdAt DESC` (для ленты) — `firestore.indexes.json` + `firebase deploy --only firestore:indexes`.

### Служебные поля монитора на задаче (пишет только сервер)

- `notifiedOverdueOn: 'YYYY-MM-DD'` — дата последней отправки «просрочено» (для «раз в день»);
- `notifiedDeadlineSoonAt: Timestamp` — одноразовое «остался 1 день»;
- `notifiedNotTakenAt: Timestamp` — одноразовое «не взял в работу за час».

Старые клиентские флаги (`reminderSent`, `overdueNotificationSent`, `lastNotTakenReminder`) остаются в данных, монитор их не читает.

## 2. `lib/telegram-send.js` (новый общий модуль)

`sendTelegramMessage(chatId, text)` с токеном из `TELEGRAM_BOT_TOKEN`. Используют: `api/notify-telegram` (рефактор), `api/agent-monitor`, ветка создания задач в `api/agent-chat`. Не serverless-функция — на лимит не влияет.

## 3. `api/agent-monitor.js` (новая функция, 11/12)

- **Авторизация:** заголовок `Authorization: Bearer ${CRON_SECRET}`. Vercel Cron шлёт его сам при заданном env `CRON_SECRET`; GitHub Actions шлёт тот же секрет из GH Secrets. Иначе 401.
- **Расписания:** `vercel.json` → `crons: [{path: '/api/agent-monitor', schedule: '0 6 * * *'}]` (~09:00 МСК, страховка) + `.github/workflows/agent-monitor.yml` → `schedule: cron('5 * * * *')` (ежечасно).
- **Логика (все даты в Europe/Moscow, не UTC):** загрузить задачи `status == 'in-progress'`, для каждой:
  - `deadline < сегодня` и `notifiedOverdueOn != сегодня` → «просрочена» исполнителю(ям) и постановщику; `notifiedOverdueOn = сегодня`;
  - `deadline == завтра` и нет `notifiedDeadlineSoonAt` → «остался 1 день» обоим; пометить;
  - `subStatus == 'assigned'`, `createdAt` старше 1 часа, нет `notifiedNotTakenAt` → «не взял в работу» обоим; пометить.
- **Доставка каждого события:** документ(ы) в `agentNotifications` + Telegram (если у получателя привязан `telegramChatId`). Получатели: `assigneeIds` (fallback `assigneeEmail`→uid) и `createdByUid`.
- Без LLM — чистая проверка дат (быстро, бесплатно, детерминированно).

## 4. Создание задач из документа — двухфазный протокол в `api/agent-chat` (без новой функции)

**Фаза 1 (предпросмотр).** Пользователь пишет агенту «сформируй задачи из документа X». Системный промпт дополняется: на такую просьбу LLM возвращает строгий JSON `{action:'propose_tasks', file, tasks:[{title, deadline:'YYYY-MM-DD', assigneeName}]}`. Сервер валидирует: матчит `assigneeName` со списком участников организации (без регистра, displayName и Имя+Фамилия; неоднозначность/не найден → пометка «не создастся» с причиной), определяет `projectId` по файлу (задачи создаются в проекте документа). Ответ клиенту: `{type:'taskProposal', proposal:[...]}`. Клиент рендерит карточку-таблицу + кнопку **«Создать N задач»**.

**Фаза 2 (подтверждение).** Кнопка шлёт `{action:'create_tasks', tasks:[...]}` на тот же endpoint. Сервер: проверка прав вызывающего (`callerCanManageProject` — owner/admin, или moderator с доступом; employee — отказ), повторная валидация, батч-создание задач Admin SDK: `status:'in-progress'`, `subStatus:'assigned'`, `createdByUid = caller`, `createdAt = serverTimestamp`, поля как при обычном создании. Затем: Telegram исполнителям + `agentNotifications` типа `tasks_created`. Ответ — сводка в чат.

## 5. UI

- **Колокольчик** в хедере рядом с «Мои задачи»: бейдж = количество непрочитанных; панель списком (иконка типа, текст, время); клик по записи → открыть задачу (существующая модалка); кнопка «прочитать все» (batch update `readAt` своих записей — разрешено правилами).
- Listener: `where('uid','==',me).orderBy('createdAt','desc').limit(50)` (композитный индекс из п.1).
- **Карточка предпросмотра** в чате агента: таблица (название/срок/ответственный/статус матчинга) + кнопка создания (видна, но сервер всё равно перепроверяет права).

## 6. Этапы реализации (каждый деплоится и проверяется отдельно)

1. **Лента:** `lib/telegram-send.js` (+рефактор notify-telegram), коллекция + правила + индекс, колокольчик UI (лента пока пустая). Тесты правил на ленту.
2. **Монитор:** `api/agent-monitor` + `vercel.json` cron + GH Actions workflow + `CRON_SECRET` (пользователь кладёт в Vercel env и GH Secrets) + отключение старых клиентских напоминаний. Юнит-тесты классификации по датам (МСК, антиспам). Смоук на тестовой задаче.
3. **Задачи из документа:** промпт + парсер/валидатор JSON + матчинг + фаза подтверждения + карточка в чате. Юнит-тесты матчинга и валидации.

Порядок деплоя каждого этапа: тесты → git push (Vercel) → правила/индексы Firebase → смоук.

## 7. Тесты

- Юниты: классификация задач по датам в Europe/Moscow (просрочена/завтра/не взял час; граничные случаи полуночи), антиспам-поля; матчинг ответственных (регистр, дубликаты, не найден); валидация JSON от LLM (мусор → отказ без создания).
- Rules-тесты: чужие уведомления не читаются; клиент не может create/delete; `readAt` обновляется только своё и только это поле.
- Смоук в проде: тестовая задача с дедлайном вчера/завтра → прогон монитора → лента + Telegram.

## 8. Риски

- Качество извлечения задач = качество документа; если текст не извлечён (`extractionStatus != 'done'`) — агент прямо говорит об этом.
- Hobby-cron плавает по времени в пределах часа; GH Actions может опаздывать на минуты — для уведомлений некритично.
- LLM может вернуть некорректный JSON — валидатор отбрасывает, пользователь видит понятную ошибку, ничего не создаётся.
- Лимит функций: станет 11/12, остаётся 1 слот.
