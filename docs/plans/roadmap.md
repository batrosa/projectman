# Desktop + Mobile App Roadmap

Дата: 2026-07-06

## Цель

Сделать HoldingMan не только web-программой, но и отдельными приложениями:

- desktop app для PC/Mac, который сотрудник скачивает один раз и использует каждый день;
- mobile app для iOS/Android с теми же аккаунтами, данными, задачами и телефонными push-уведомлениями.

## Позиционирование продукта

HoldingMan — система контроля проектов, задач и сроков для холдингов.

Ключевая идея: это не перегруженный CRM-комбайн для всех процессов бизнеса, а рабочий инструмент руководителя и команды для ежедневного контроля исполнения.

Фокус продукта:

- проекты;
- задачи;
- ответственные;
- сроки;
- статусы работ;
- контроль исполнения;
- риски срыва сроков;
- роли и доступы;
- уведомления;
- Гант;
- ИИ-агент как помощник руководителя.

Отстройка от Bitrix24 и похожих систем:

- проще внедрить;
- дешевле владеть и оплачивать;
- меньше лишних модулей;
- меньше перегруза для сотрудников;
- быстрее понять и начать использовать;
- фокус именно на проектах, задачах и холдинговой структуре;
- ИИ-агент помогает руководителю ставить задачи, отслеживать сроки и видеть проблемы.

Базовая формулировка для landing:

> HoldingMan помогает руководителю холдинга контролировать проекты, задачи, сроки и исполнителей в одном месте — без перегруженных CRM-модулей и сложного внедрения.

Короткий слоган:

> Проекты, задачи и сроки холдинга под контролем.

## Принцип архитектуры

Все клиенты должны работать с одной общей backend-средой:

- Firebase Auth — единые аккаунты пользователей;
- Firestore — единые организации, проекты, задачи, роли, уведомления;
- Vercel API — единая серверная логика агента, файлов, мониторинга и уведомлений;
- текущая web-версия — остается основным источником UI/логики для desktop-оболочки.

Electron, iOS и Android не должны синхронизироваться друг с другом напрямую. Они должны подключаться к тем же backend-сервисам.

## Этап 1 — Desktop App для PC/Mac

Рекомендуемый путь: Electron.

Требования:

- отдельное устанавливаемое приложение для Windows;
- отдельное устанавливаемое приложение для macOS;
- сохранить текущий визуал, дизайн и функции максимально близко к 100%;
- использовать текущую web-версию HoldingMan внутри desktop-оболочки;
- сотрудник скачивает приложение один раз, дальше работает через интернет;
- приложение открывает актуальную production-версию HoldingMan;
- авторизация, задачи, проекты, уведомления, агент и роли работают через те же Firebase/Vercel сервисы.

Что нужно реализовать:

- Electron shell;
- app icon и название;
- загрузка production URL;
- обработка внешних ссылок;
- отдельные сборки для macOS и Windows;
- code signing/notarization для macOS;
- Windows installer;
- автообновления или простой механизм обновления desktop-оболочки;
- проверка Firebase Auth/Telegram login/file upload внутри Electron.

## Этап 2 — Mobile App для iOS/Android

Варианты:

- быстрый и максимально близкий к текущему интерфейсу путь: WebView/Capacitor;
- более качественный нативный путь: Swift/SwiftUI для iOS и отдельный Android-клиент.

Принятое понимание:

- если писать iOS на Swift, данные и аккаунты сохраняются полностью через тот же backend;
- текущий web-визуал 100% пиксель-в-пиксель в Swift не сохраняется, потому что UI-система другая;
- можно сохранить бизнес-логику, данные, роли и функции, но мобильный интерфейс будет нативной адаптацией.

Требования:

- один аккаунт пользователя на web/desktop/iOS/Android;
- одни и те же организации, проекты, задачи, статусы, роли и уведомления;
- мобильный UI должен учитывать iOS/Android UX;
- все критичные операции должны идти через те же правила доступа и серверные проверки;
- не дублировать бизнес-логику небезопасно на клиенте, если она уже должна быть серверной.

## Этап 3 — Push-уведомления на телефон

Цель: уведомления, которые сейчас есть в программе, должны приходить на iPhone/Android напрямую как системные push.

События для push:

- новая задача;
- задача возвращена на доработку;
- задача просрочена;
- остался 1 день до дедлайна;
- задача не взята в работу;
- будущие agentNotifications, если они критичны для пользователя.

Рекомендуемая архитектура:

```text
Событие в HoldingMan
  -> сервер создает agentNotification
  -> сервер находит device tokens пользователя
  -> сервер отправляет push через FCM/APNs
  -> iOS/Android показывает системное уведомление
```

Что нужно реализовать:

- iOS app регистрируется в Apple Push Notification service;
- Android app регистрируется в Firebase Cloud Messaging;
- mobile app получает push/device token;
- token сохраняется в Firestore, например `users/{uid}/devices/{deviceId}`;
- сервер отправляет push на все активные устройства пользователя;
- протухшие/ошибочные tokens удаляются или помечаются неактивными;
- при logout token отвязывается от пользователя;
- уведомление должно открывать нужную задачу/проект в приложении.

Инфраструктура:

- Apple Developer Account;
- iOS Bundle ID;
- Push Notifications capability в Xcode;
- APNs key/certificate;
- Firebase iOS app в текущем Firebase project;
- Firebase Android app в текущем Firebase project;
- Firebase Cloud Messaging;
- серверная отправка push через Firebase Admin SDK или прямой APNs/FCM.

## Этап 4 — Усиление безопасности

Цель: максимально проработать безопасность программы в части доступа, утечек данных, взломов, подмены ролей и несанкционированных действий.

Основные направления:

- безопасность авторизации;
- защита ролей и прав доступа;
- изоляция организаций друг от друга;
- защита серверных API;
- защита Firestore rules;
- защита файлов и базы знаний проекта;
- защита agent-функций от опасных действий без подтверждения;
- аудит логов и расследование инцидентов;
- защита от утечек персональных данных и служебных токенов;
- безопасная работа desktop/mobile клиентов.

Что нужно проверить и реализовать:

- полный аудит Firestore rules по всем коллекциям;
- запрет любых client-side изменений ролей, `organizationId`, `allowedProjects`, XP/statistics и системных полей;
- серверная проверка всех мутаций через Admin SDK;
- строгая tenant isolation: пользователь не должен читать/менять данные чужой организации;
- проверка всех API на IDOR-уязвимости: нельзя передать чужой `projectId`, `taskId`, `uid`, `orgId` и получить доступ;
- rate limits для agent/API endpoints;
- аудит опасных действий: удаление задач, удаление организации, изменение ролей, исключение пользователей, загрузка/удаление файлов;
- immutable audit log для критичных действий;
- ограничение загрузки файлов по размеру, типу и содержимому;
- антивирусная/контентная проверка файлов, если появится публичная загрузка;
- защита от XSS в web/desktop/mobile UI;
- защита от prompt injection в файлах проекта и ответах агента;
- запрет агенту выполнять destructive actions без preview + подтверждения;
- хранение секретов только в server env, не в клиенте;
- ротация секретов и ключей;
- настройка CSP/security headers;
- проверка Firebase Auth providers и lifecycle logout/session revoke;
- мониторинг подозрительных действий;
- регулярный security regression suite.

Отдельно для desktop/mobile:

- secure storage для локальных токенов;
- запрет сохранения секретов в localStorage, если появится native-клиент;
- проверка deep links на подмену параметров;
- защита push tokens;
- logout должен отзывать/отвязывать device token;
- code signing для desktop app;
- notarization для macOS;
- защита auto-update канала.

Основные страницы:

- главная landing-страница;
- тарифы;
- страница скачивания приложения для Windows/macOS;
- страница мобильных приложений iOS/Android;
- страница входа/перехода в web-приложение;
- политика конфиденциальности;
- условия использования;
- документы по оплате/возвратам;
- контакты поддержки.

Функции landing:

- описание продукта и преимуществ;
- демонстрация интерфейса;
- блоки для руководителей, сотрудников и организаций;
- тарифные планы;
- кнопка оплаты подписки;
- кнопка скачивания desktop app;
- ссылки на App Store / Google Play после появления mobile app;
- форма заявки/контакта;
- FAQ;
- юридические документы.

Оплата и подписки:

- выбрать платёжного провайдера;
- подключить приём платежей;
- привязать расчётный счёт/аккаунт получателя платежей;
- настроить тарифы;
- настроить recurring subscriptions;
- обрабатывать успешную оплату через webhook;
- обрабатывать отмену подписки;
- обрабатывать просрочку платежа;
- хранить subscription status в backend;
- связать оплату с организацией в HoldingMan;
- ограничивать доступ организации при неоплаченной подписке;
- предусмотреть trial-период;
- предусмотреть ручную выдачу доступа администратором;
- сделать страницу управления подпиской;
- сделать счета/чеки/закрывающие документы, если это требуется юридически.

Техническая схема:

```text
Landing
  -> выбор тарифа
  -> платёжный провайдер
  -> webhook оплаты
  -> backend обновляет subscription status организации
  -> HoldingMan разрешает или ограничивает доступ
```

Что нужно решить отдельно:

- юридическое лицо или ИП, на кого принимать оплату;
- страна/валюта платежей;
- платёжный провайдер;
- нужна ли онлайн-касса/чеки;
- нужны ли договоры и акты для B2B;
- политика возвратов;
- лимиты тарифов: количество пользователей, проектов, файлов, agent-запросов;
- поведение при неоплате: read-only, блокировка или grace period.

### Реализация оплаты «за пользователя в месяц» (seat-модель)

> Важно: платёжную часть нельзя запускать без юридической проверки (юрлицо/ИП,
> налоги, чеки по 54-ФЗ, возвраты, оферта). Сначала юридическая модель — потом код.

**Кто и за что платит.** Платит организация (её владелец), а не каждый сотрудник
сам. Расчётная единица — «место» (seat):

```text
сумма за период = число участников организации × цена за место (напр. 390 ₽)
```

Уже есть `organizations/{orgId}.membersCount` — это и есть фактические места.
Сотрудники ничего не оплачивают; модель «каждый платит за себя» для холдингов
не рассматриваем.

**Провайдер (РФ, рубли, рекуррент).** ЮKassa (автоплатежи + встроенная
фискализация 54-ФЗ), альтернативы — CloudPayments / Тинькофф Касса / Продамус.
Рекуррент: первый платёж с сохранением способа оплаты → `payment_method_id` →
дальше сервер списывает сам каждый период (карта хранится у провайдера, у нас —
только токен).

**Модель данных.** Подписка живёт на организации:

```js
organizations/{orgId}.subscription = {
  status: 'trialing' | 'active' | 'past_due' | 'canceled',
  plan: 'team',
  seats: 10,               // оплачено мест
  pricePerSeat: 390,
  currentPeriodEnd: <ts>,
  paymentMethodId: '<токен провайдера>'
}
```

**Жизненный цикл + контроль доступа (обязательно серверный).**

```text
trial 14 дней → active → (списание) → продлено
                       ↘ неоплата → past_due (grace 3–5 дней) → read-only
```

Проверка статуса — на сервере (Firestore rules + Vercel API читают
`subscription.status`), как вся остальная безопасность. При неоплате — read-only,
данные клиента НЕ удалять. Новый endpoint `api/billing`: создать платёж + принять
webhook провайдера (проверка подписи → обновить `subscription`).

### Контроль числа мест: «оплатили 10, пригласили 11-го»

Ключевое: **вступление уже серверное** (`api/join-org`, Admin SDK). Приглашение по
коду само по себе членства не даёт — его выдаёт сервер, клиент не может присвоить
себе `organizationId`. Значит, лимит мест проверяется в одной защищённой точке, и
«молча» превысить его нельзя:

```text
вступление по коду → api/join-org → проверка: membersCount < subscription.seats?
   да  → добавить участника (транзакция на membersCount закрывает гонку)
   нет → отказ / доплата
```

То же — при прямом добавлении участника и, при желании, смене роли (`api/org`).

**Три стратегии на 11-го участника:**

1. **Жёсткий лимит (проще, без сюрпризов).** 11-й получает отказ: «Оплачено 10
   мест — увеличьте тариф». Владелец сначала докупает место. Рекомендуется для v1.
2. **Авто-расширение + доплата (как Slack/Notion).** 11-й вступает, `seats` растёт
   до 11, доплата за место списывается сразу (пропорционально остатку периода) или
   в следующий счёт. Лучший UX, но требует рекуррента И явного согласия в оферте
   (иначе «сюрприз-списание» — так нельзя).
3. **Постоплата / сверка (для юрлиц по счёту).** Вступают свободно, в конце периода
   счёт по фактическому числу участников. Просто, честно, подходит холдингам на
   безнале, но оплата постфактум.

Рекомендация: для холдингов — стратегия 3 (сверка) или 1 (жёсткий лимит), обе без
карточного рекуррента и без сюрприз-списаний. Авто-доплату (2) добавлять позже и
только с явным согласием владельца.

**Нюансы к решению:**

- что считается местом: все участники или только активные роли (наблюдателей/reader
  можно сделать бесплатными);
- освобождение места: убрали сотрудника — слот свободен до конца оплаченного
  периода, без возврата за середину месяца;
- участник занимает слот в момент вступления (по `membersCount`), а не когда
  «начнёт работать»;
- на trial тоже ограничить максимум мест по тарифу.

**Точки в коде (когда дойдём до реализации):** проверка лимита в `api/join-org`
(вступление по коду), в `api/org` (прямое добавление участника; при необходимости —
`updateMemberRole`), новый `api/billing` (платёж + webhook), поле `subscription` на
организации и его чтение в Firestore rules / серверных проверках доступа.

## Этап 6 — Интеграция с 1С

Цель: дать клиентам, купившим подписку, понятный и безопасный способ подключить свою 1С к HoldingMan, чтобы данные из 1С попадали в проекты, задачи, финансовые статусы и уведомления.

Важно: 1С у клиентов может быть разной:

- 1С:Бухгалтерия;
- 1С:Управление торговлей;
- 1С:ERP;
- 1С:ЗУП;
- 1С:Документооборот;
- кастомные или сильно доработанные конфигурации.

Поэтому нельзя обещать универсальное подключение к любой 1С за 5 минут. Правильная модель:

> Для типовых конфигураций — готовый коннектор. Для доработанных 1С — настройка маппинга или внедрение под ключ.

### Модель внедрения для клиента

Клиентский сценарий:

1. Клиент регистрируется в HoldingMan.
2. Создает организацию.
3. Покупает подписку.
4. Открывает `Настройки -> Интеграции -> 1С`.
5. Создает integration token.
6. Скачивает 1С-коннектор.
7. Передает коннектор своему 1С-специалисту или заказывает внедрение у нас/партнера.
8. В 1С указываются API URL и integration token.
9. Выбираются данные для синхронизации.
10. Запускается тест соединения.
11. Включается расписание синхронизации.
12. В HoldingMan появляются связанные данные из 1С.

### Варианты подключения

#### Вариант 1 — ручная настройка

В HoldingMan:

- кнопка `Создать ключ интеграции`;
- кнопка `Скачать обработку для 1С`;
- инструкция;
- кнопка `Проверить соединение`;
- журнал обмена.

В 1С:

- устанавливается внешняя обработка или расширение;
- вставляется URL HoldingMan;
- вставляется integration token;
- выбираются справочники и документы;
- запускается тестовый обмен.

Подходит для небольших клиентов, где есть 1С-специалист.

#### Вариант 2 — официальный 1С-коннектор

Сделать продуктовый модуль `HoldingMan Connector for 1C`.

Форматы:

- внешняя обработка `.epf`;
- расширение `.cfe`;
- отдельная поставка под типовые конфигурации.

Функции коннектора:

- ввод API URL;
- ввод integration token;
- выбор организации HoldingMan;
- выбор справочников и документов для обмена;
- расписание синхронизации;
- журнал ошибок;
- повтор отправки при сбое;
- кнопка `Отправить тестовые данные`;
- HMAC-подпись или token auth;
- защита от повторной отправки одного события.

Это основной коммерческий вариант.

#### Вариант 3 — внедрение под ключ

Для крупных холдингов:

- анализ конфигурации 1С;
- настройка маппинга;
- настройка прав доступа;
- настройка регламентных заданий;
- тестирование обмена;
- обучение сотрудников;
- сопровождение запуска.

Можно продавать отдельно как onboarding/integration service.

### Данные из 1С

Первая версия интеграции должна быть read-only из 1С в HoldingMan.

Потенциальные данные:

- сотрудники;
- подразделения;
- организации/юрлица;
- контрагенты;
- договоры;
- счета;
- оплаты;
- задолженность;
- акты;
- заказы клиентов;
- заказы поставщикам;
- проекты/объекты;
- статьи затрат;
- план/факт затрат;
- статусы документов;
- даты оплат/отгрузок/закрытия документов.

Наиболее полезно для HoldingMan:

- создавать проекты из договоров, заказов или объектов 1С;
- подтягивать сотрудников и подразделения;
- показывать в проекте сумму договора, оплачено, долг, план/факт;
- создавать задачи по событиям 1С;
- связывать задачу HoldingMan с документом 1С;
- показывать руководителю не только статус задачи, но и финансовый риск.

Примеры автоматических задач:

- счет не оплачен;
- нет закрывающих документов;
- просрочена поставка;
- договор без ответственного;
- акт не подписан;
- задолженность превысила лимит;
- проект есть в 1С, но нет ответственных в HoldingMan.

### Техническая схема

```text
1С
  -> HoldingMan Connector for 1C
  -> /api/1c-sync
  -> серверная проверка token/HMAC
  -> Firestore
  -> проекты / задачи / финансовые статусы / уведомления
```

Основные сущности в HoldingMan:

- `integrations/{integrationId}`;
- `integrationSyncLogs/{logId}`;
- `projects[].1cExternalId`;
- `tasks[].1cExternalId`;
- `users[].1cExternalId`;
- `organizations[].subscriptionStatus`;
- отдельные коллекции или поля для договоров, счетов, оплат и задолженности.

### Что нужно разработать в HoldingMan

- раздел `Интеграции`;
- карточка интеграции `1С`;
- генерация integration token;
- безопасное хранение hash токена;
- endpoint `api/1c-sync`;
- проверка подписи или token auth;
- журнал синхронизации;
- экран ошибок синхронизации;
- маппинг 1С -> HoldingMan;
- защита от дублей через `1cExternalId`;
- права: подключать 1С может только owner/admin;
- тарифное ограничение: интеграция 1С только на нужном плане;
- уведомления при сбое синхронизации;
- ручной запуск синхронизации;
- отключение интеграции.

### Что нужно разработать в 1С

- обработка/расширение;
- форма настроек подключения;
- выбор данных для обмена;
- регламентное задание;
- журнал обмена;
- повтор отправки при ошибке;
- отправка JSON в HoldingMan;
- HMAC-подпись или token auth;
- тест соединения;
- настройка периода синхронизации.

### Безопасность 1С-интеграции

- не подключаться напрямую к базе 1С;
- не открывать 1С в интернет без необходимости;
- предпочтительно: исходящий HTTPS-запрос из 1С/коннектора в HoldingMan;
- отдельный integration token на организацию;
- возможность отозвать token;
- HMAC-подпись запросов;
- HTTPS only;
- IP allowlist или VPN для крупных клиентов;
- лог всех обменов;
- минимальный набор данных;
- запрет записи обратно в 1С на первой версии;
- защита от replay attack;
- ограничение частоты запросов;
- отдельные права на включение/отключение интеграции.

### Рекомендуемый порядок реализации

1. Спроектировать модель `integrations` и `1cExternalId`.
2. Добавить раздел `Интеграции -> 1С` в HoldingMan.
3. Реализовать `api/1c-sync` только для read-only приема данных.
4. Сделать первый 1С-коннектор для типовой конфигурации.
5. Начать с данных: сотрудники, контрагенты, договоры, счета, оплаты, задолженность.
6. Добавить журнал синхронизации и экран ошибок.
7. Добавить автоматическое создание задач по событиям 1С.
8. Протестировать на демо-базе 1С.
9. Подготовить инструкцию для клиента и 1С-специалиста.
10. После стабильной read-only версии отдельно рассмотреть обратную запись статусов в 1С.

## Важные ограничения

- Push на iOS требует разрешения пользователя.
- Focus/Do Not Disturb на телефоне может скрывать звук или баннер.
- Apple/Google не гарантируют мгновенную доставку в 100% случаев.
- Desktop Electron не является backend и не должен быть связующим звеном для mobile app.
- Для 100% сохранения web-визуала на desktop нужен Electron; для Swift/iOS это будет нативная адаптация, а не точная копия HTML/CSS.
- Платёжную часть нельзя запускать без юридической проверки провайдера, налогов, чеков, возвратов и правил подписки.
- Безопасность должна проверяться перед каждым публичным релизом, особенно перед запуском платных подписок.
- Интеграцию с 1С нельзя обещать как универсальную для любой кастомной конфигурации без предварительного анализа.
- Первая версия 1С-интеграции должна быть read-only из 1С в HoldingMan; обратную запись в 1С делать позже и только после отдельного аудита.

## Предварительный порядок работ

Рекомендуемая последовательность важна: сначала закрываем безопасность и коммерческую основу, потом быстро выпускаем desktop-приложение как самый близкий к текущей web-версии продукт, затем mobile/push, и только после этого сложные B2B-интеграции вроде 1С.

### Фаза 0 — Упаковка текущего продукта

Цель: привести текущий HoldingMan к состоянию, которое не стыдно показывать первым клиентам.

1. Зафиксировать позиционирование: проекты, задачи, сроки, холдинг, проще и дешевле перегруженных CRM-комбайнов.
2. Стабилизировать текущий web-продукт.
3. Проверить основные сценарии: организация, роли, проекты, задачи, Гант, календарь, агент, уведомления, Telegram.
4. Зафиксировать демо-сценарий для продажи.

### Фаза 1 — Безопасность перед публичным масштабированием

Цель: не запускать платный продукт и внешних клиентов без проверки доступа, ролей и утечек.

1. Провести отдельный security audit.
2. Закрыть риски Firestore rules.
3. Проверить все server API на IDOR и обход ролей.
4. Проверить файлы, agent-функции и destructive actions.
5. Настроить audit logs для критичных действий.
6. Настроить security headers/CSP.
7. Добавить regression-тесты на найденные риски.

### Фаза 2 — Desktop app для PC/Mac

Цель: быстро дать пользователям отдельное приложение, скачиваемое с сайта, почти полностью сохранив текущий визуал и функции.

1. Подготовить Electron desktop app для Mac/Windows.
2. Подключить production URL HoldingMan.
3. Проверить авторизацию, файлы, чат агента, уведомления и Telegram внутри Electron.
4. Настроить app icon, название, окно, внешние ссылки.
5. Настроить сборки для macOS и Windows.
6. Настроить code signing/notarization для macOS.
7. Настроить Windows installer.
8. Настроить auto-update или простой механизм обновления оболочки.
9. Протестировать ежедневное использование сотрудником.

### Фаза 3 — Landing + подписки

Цель: сделать коммерческую точку входа: объяснить продукт, дать скачать приложение и принять оплату.

1. Спроектировать тарифы и subscription model.
2. Решить юридическую модель оплаты: юрлицо/ИП, страна, валюта, чеки, возвраты.
3. Реализовать продающий landing.
4. Добавить страницы тарифов, скачивания desktop app, политики и условий.
5. Подключить платёжного провайдера.
6. Подключить payment webhooks.
7. Связать оплату с организациями в HoldingMan.
8. Реализовать trial/grace period/ограничение доступа при неоплате.
9. Протестировать оплату, отмену, просрочку и восстановление подписки.

### Фаза 4 — Push-инфраструктура

Цель: подготовить backend для телефонных уведомлений до выпуска mobile app.

1. Добавить device token модель в Firestore.
2. Добавить привязку token к пользователю и организации.
3. Добавить logout/unregister token.
4. Добавить серверную отправку push из существующих мест создания уведомлений.
5. Обрабатывать протухшие и ошибочные tokens.
6. Добавить логи доставки push.
7. Протестировать безопасность push tokens.

### Фаза 5 — Mobile app iOS/Android

Цель: дать мобильный доступ к тем же аккаунтам, задачам, проектам и уведомлениям.

1. Принять техническое решение: WebView/Capacitor для быстрого запуска или native Swift/Android для долгосрочного качества.
2. Реализовать вход в тот же Firebase Auth.
3. Подключить те же Firestore/Vercel API.
4. Реализовать основные сценарии: мои задачи, проекты, статусы, уведомления, задача, дедлайн.
5. Подключить системные push-уведомления.
6. Реализовать deep links: push открывает конкретную задачу/проект.
7. Протестировать права, роли, мультиустройства, logout и удаление tokens.
8. Подготовить сборки для TestFlight / Google Play testing.

### Фаза 6 — 1С-интеграция

Цель: сделать B2B-функцию для клиентов с 1С, но не начинать с нее, потому что конфигурации 1С сильно отличаются.

1. Спроектировать интеграцию с 1С и модель `1cExternalId`.

3. Реализовать `api/1c-sync` только для read-only приема данных.

6. Протестировать обмен с демо-базой 1С.
7. Добавить журнал синхронизации и экран ошибок.
