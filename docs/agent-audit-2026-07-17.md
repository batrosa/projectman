# Аудит ИИ-агента HoldingMan — 17.07.2026

Проверка четырёх подсистем: ядро чата (`api/agent-chat.js`), автопилот (`agent-monitor`), файлово-знаковый пайплайн, UX и документация. Тесты: 428/428 зелёные. Аудит по коду, не по комментариям; все находки верифицированы чтением, ключевые — воспроизведением.

## Общий вердикт

Фундамент необычно зрелый: мутации детерминированы и идут мимо LLM (preview → confirm → серверное исполнение), идемпотентность через `agentProposalExecutions`, транзакционный claim антиспам-флагов, защита от system-role-инъекций, XSS-рендер через textContent. Записи ломать модель не может.

Главная проблема в другом: **агент формально grounded, но фактически слеп** — он не знает текущую дату, не видит истории задач, а база знаний ретривится наивным substring-матчингом. Плюс каждое «привет» перечитывает всю организацию с полными текстами файлов. Это то, что отделяет агента от «действительно полезного».

## 2. Средние проблемы (MEDIUM, выборочно)

**Ядро агента:**
- Имена сущностей не санитизируются перед `<holdingman_untrusted_data>` — проект с именем `</holdingman_untrusted_data>…` ломает границу доверия (:457, 3235-3304)
- `hasFalseExecutionClaim` не ловит «добавил», «выполнено», английские формулировки (:3443-3456)
- Идемпотентность только у create/delete tasks и create_project; rename/take/delete_project — нет (:2362-2555)
- Confirm-хендлеры грузят до 200 документов последовательно — риск 504 (:2299-2329); фикс: `db.getAll`
- Нет org-level бюджета LLM и учёта токенов; delete_notification обходит rate-limiter (:180-185)
- В retrieval подмешаны 4 последние реплики истории — тянут нерелевантные чанки (:352)
- В крупной организации knowledge-бюджет ~13.5k = 2-3 чанка на все файлы

**Монитор:**
- LLM-совет по дизайну валидатора не может содержать цифр/дат/имён → ценность ≈ шаблон, но платим латентностью и деньгами (`agent-monitor-ai.js:25-35`)
- Флаг антиспама выставляется до факта доставки — потери Telegram 429 молчаливы
- `unassigned_1h` уходит только создателю, одноразово — нет эскалации админам
- Часовая чувствительность целиком зависит от GitHub Actions (который выбрасывает запуски); heartbeat-контроля нет
- Прогон последовательный при maxDuration 60s — не масштабируется

**Файлы:**
- Скан-PDF → `extractionStatus:"error"` без объяснения (самый частый B2B-кейс)
- Даты xlsx попадают в базу знаний как Excel-сериалы (46179)
- Таблицы docx теряют структуру (`</w:tc>` не заменяется)
- `agent-task-file.js`: нет rate-limit'а и нет untrusted-обёртки содержимого файла (prompt injection вектор)
- «Удалённый» файл живёт в Cloudinary вечно (unsigned preset, нет server-side delete)
- Застрявшая экстракция без retry; ре-аплоад дублирует чанки

**UX:**
- Нет кнопки «Отмена» ни на одной карточке (web + iOS); на web после подтверждения кнопка просто исчезает (нет done-стейта)
- Пустой чат не учит возможностям — 80% функций (take_task, rename/delete project, навигация) пользователь не может узнать из продукта
- Нет retry на ошибках; история чата не персистентна

## 3. Документация vs реальность

- §15 overview: cron `0 6 * * *` → фактически `0 15 * * *` + почасовой GitHub Actions
- **Прямое противоречие**: `mass-launch-readiness.md` требует называть имя файла в ответах, а SYSTEM_PROMPT требует НЕ называть (`agent-chat.js:~93`)
- iOS-раздел занижает возможности (iOS уже поддерживает все actionProposal)
- P1-к-запуску не реализованы: эталонные 100 вопросов, кнопки 👍/👎, org-лимиты стоимости, entitlement-проверка тарифа (агент сейчас доступен всем — первое место, которое забудут закрыть при монетизации)

## 4. Чего не хватает для настоящего «AI project manager»

По убыванию ценности, всё реализуемо в текущей архитектуре:

1. **Утренний дайджест-планёрка (9:05 МСК)** — «Сегодня срок по N задачам, просрочено M» одним сообщением на пользователя
2. **Детекция зависших задач** — `in_work` без прогресса N дней; `completed` (на проверке) старше 24-48ч → пинок постановщику
3. **Эскалационные цепочки** — unassigned повтор через 24ч + админы; overdue >3 дней → админы; возраст просрочки в тексте
4. **Per-file авто-summary** при загрузке (3-5 предложений + ключевые даты/суммы) — ответы «о чём документ» и маршрутизация к файлу
5. **Анализ загрузки** — «У Ивана 12 активных, 3 просрочены» в дайджесте/при назначении
6. **Счётчики активности в контекст чата** — created/completed за 7/30 дней на проект
7. **Еженедельный дайджест для owner/admin**, настройки quiet hours, snooze уведомлений

## 5. Приоритизированный план (топ-15)

**Быстрые победы (1-2 дня, максимальный эффект):**
1. Текущая дата + день недели в основной промпт (1.1)
2. Фикс маршрута «дай список/перечень/отчёт» + тест (1.4)

4. Окно тишины для `not_taken_1h` + событие `deadline_today` утром с catch-up (1.7, 1.8)
5. Показ 429 с текстом сервера на web (1.11)
6. Чипы-примеры в пустом чате (web + iOS)

**Ядро полезности (неделя):**

9. Агрегация уведомлений по получателю + утренний дайджест (1.9)
10. Каскадное удаление files при delete_project + TTL для agentProposalExecutions/agentNotifications (1.10)

12. Структурные фиксы извлечения: docx `</w:tc>`, xlsx-даты→ISO, детект скан-PDF, untrusted-обёртка + rate-limit в agent-task-file

**Платформа полезности (2-3 недели):**
