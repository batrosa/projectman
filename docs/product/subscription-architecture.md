# Подписки HoldingMan: тарифы и защищённая архитектура

Статус: продуктовая и техническая спецификация. Оплата и тарифные блокировки в production пока не включены. Тарифы показываются на лендинге; рабочее веб-приложение и iOS-клиент до отдельного этапа продолжают работать по текущим правилам.

## Тарифная сетка

| Тариф | Активные места | ИИ-агент | Цена в месяц |
|---|---:|---:|---:|
| Free | до 5 включительно | нет | 0 ₽ |
| Team 50 | до 50 включительно | да | 4 990 ₽ |
| Business 100 | до 100 включительно | да | 9 990 ₽ |
| 100+ | 101+ | да | индивидуально; в разработке |

Место — активное членство в организации, включая владельца. Неиспользованное приглашение резервирует место до истечения срока: иначе можно одновременно выдать много приглашений и превысить лимит при параллельном вступлении.

Предложенная сетка понятна и годится для запуска. На верхней границе Team 50 и Business 100 цена почти одинаковая — около 100 ₽ за человека в месяц. Это выгодно крупным командам, но компании из 6–15 человек будут платить заметно больше на одно место. Перед включением оплаты стоит проверить спрос; возможный промежуточный тариф до 15–20 человек снизит этот барьер. Годовой вариант можно дать со скидкой в два месяца: 49 900 ₽ и 99 900 ₽, но это продуктовая гипотеза, а не обязательная часть архитектуры.

## Где показывается и покупается тариф

- Публичные карточки тарифов находятся на лендинге HoldingMan.
- Выбор и оплата тарифа в будущем происходят на отдельной защищённой странице сайта после входа владельца организации.
- В рабочем веб-приложении можно позднее показывать только текущий тариф, лимит и статус оплаты; платёжная форма там не обязательна.
- iOS остаётся бесплатным companion-клиентом существующего B2B-аккаунта: без покупки, ссылки «оплатить» и призыва перейти на внешний платёж. Это снижает риск требования Apple In-App Purchase по App Review Guideline 3.1.3(f).
- Сайт не доверяет переданному из браузера `organizationId`: сервер получает владельца из Firebase ID token и проверяет его активное членство и роль owner.

## Рекомендуемый платёжный контур

Для рублёвых платежей базовый кандидат — ЮKassa. Она поддерживает сохранение способа оплаты и последующие автоплатежи, но production-автоплатежи подключаются через менеджера ЮKassa. До заключения договора HoldingMan не должен обещать автоматическое продление как уже работающую функцию.

Поток первой оплаты:

1. Владелец на сайте выбирает тариф.
2. `POST /api/billing/checkout` проверяет роль owner, текущий тариф и допустимость перехода.
3. Сервер создаёт платёж у провайдера с уникальным idempotence key, `save_payment_method: true` и return URL сайта.
4. Браузер переходит на подтверждение ЮKassa.
5. Возврат пользователя на сайт показывает только «платёж обрабатывается»; он не активирует тариф.
6. Подписку активирует только проверенный webhook `payment.succeeded`.
7. Идентификатор сохранённого способа оплаты хранится зашифрованно/серверно и не выдаётся клиентам.

Ежемесячное продление:

1. Планировщик выбирает подписки, срок которых подходит к продлению.
2. Для каждой создаёт автоплатёж с новым idempotence key.
3. Успех продлевает `currentPeriodEnd` строго на один период.
4. Неуспех переводит подписку в `past_due`, уведомляет владельца и запускает заранее определённый grace-период.
5. Повторные попытки ограничены; после grace статус становится `suspended`.
6. Отмена запрещает следующие списания, но сохраняет оплаченные возможности до конца периода.

Обязательные серверные endpoints будущей реализации:

```text
GET  /api/billing/plans            публичная неизменяемая сетка тарифов
GET  /api/billing/subscription     текущий безопасный статус организации
POST /api/billing/checkout         только owner, создание первой оплаты
POST /api/billing/change-plan      только owner, upgrade/downgrade policy
POST /api/billing/cancel           только owner
POST /api/billing/webhook/yookassa подпись/аутентичность, идемпотентная обработка
POST /api/billing/renew            только защищённый cron/service token
```

## Единственный источник прав

Права подписки хранит сервер в `organizationEntitlements/{organizationId}`:

```text
planCode: free | team50 | business100 | enterprise
seatLimit: 5 | 50 | 100 | индивидуальное число
aiEnabled: boolean
status: trialing | active | grace | past_due | suspended | canceled
currentPeriodEnd: timestamp
providerCustomerId: string
providerSubscriptionId: string
version: number
updatedAt: timestamp
```

Дополнительные коллекции:

```text
billingEvents/{providerEventId}  // идемпотентность webhook
billingPayments/{paymentId}      // сумма, валюта, статус, период, orgId
seatCounters/{organizationId}    // activeSeats, reservedSeats, version
seatReservations/{reservationId}// TTL для одноразового приглашения
```

Клиенты web/iOS могут получить только безопасное представление тарифа для интерфейса. Они не пишут тариф, лимит, статус или флаг ИИ. Firestore rules запрещают прямую запись. Custom claims допустимы только как кеш для интерфейса, но не как источник решения: они обновляются не мгновенно.

## Защита лимита мест

Все пути добавления человека — код приглашения, ссылка, восстановление членства, админское добавление и импорт — вызывают одну серверную операцию `reserveOrCreateSeat`.

Операция выполняется одной Firestore-транзакцией:

1. Читает entitlement организации.
2. Читает серверный счётчик активных и зарезервированных мест.
3. Проверяет `activeSeats + reservedSeats < seatLimit`.
4. Создаёт резерв/членство и увеличивает счётчик в той же транзакции.
5. При конфликте транзакция повторяется; 51-й пользователь тарифа Team 50 всегда получает отказ.

`membersCount` в клиентском документе не является основанием для допуска. Его можно хранить как проекцию для интерфейса и периодически сверять с `organizationMemberships`. Удаление участника и истечение приглашения освобождают место транзакционно.

Понижение тарифа запрещено, если занято или зарезервировано больше мест, чем допускает новый тариф. Альтернатива — запланировать понижение на конец периода и потребовать сначала освободить места. Уже состоящих участников нельзя молча удалять.

## Защита ИИ-агента

Проверка выполняется в каждом серверном endpoint ИИ до обращения к модели и до расходования платных токенов:

```text
authenticated user
  -> active organization membership
  -> entitlement.status permits service
  -> entitlement.aiEnabled == true
  -> organization/user rate limit
  -> agent operation
```

Скрыть кнопку на Free недостаточно. `/api/agent-chat` и `/api/agent-task-file` обязаны сами возвращать `upgrade_required`; прямой HTTP-запрос не должен обходить тариф. При `past_due` рекомендуется короткий grace-период только для чтения обычных задач, а ИИ отключать по заранее определённому правилу.

## Оплата и смена тарифа

- Webhook платёжного провайдера проверяет подпись на необработанном теле запроса.
- Каждый event ID записывается и применяется один раз: повторная доставка не продлевает подписку повторно.
- Только webhook/внутренняя админ-операция меняют entitlement. Ответ браузера «оплата успешна» ничего не активирует.
- Переходы статусов задаются конечным автоматом; неизвестный или старый webhook не может откатить более новое состояние.
- Ручное изменение тарифа сотрудником поддержки требует причины и записи в `auditLogs`.
- Provider IDs и платёжные данные не выдаются клиентам.

## Безопасный запуск

1. Создать entitlements и закрыть их правилами Firestore.
2. Заполнить entitlement всем существующим организациям без немедленной блокировки.
3. Включить теневую проверку: логировать, кого лимит заблокировал бы, но пока пропускать.
4. Исправить расхождения членств и счётчиков.
5. Включить серверную блокировку новых мест.
6. Подключить тарифную проверку ИИ за feature flag.
7. После проверки оплаты включить реальное списание и автоматическую смену статуса.

Feature flags включаются раздельно:

```text
billingUiEnabled            // страница оплаты на сайте
seatLimitShadowMode         // только логирование превышений
seatLimitEnforced           // реальная блокировка нового места
agentEntitlementShadowMode  // логирование вызовов ИИ на Free
agentEntitlementEnforced    // реальная блокировка ИИ на Free
autoRenewEnabled            // автоматические повторные платежи
```

Так можно сначала проверить реальные данные и оплату, не отключив существующим организациям доступ внезапно.

Нельзя сразу заменить текущий `settings.maxUsers: 100` на 5: существующие организации и тестовые данные могут оказаться выше лимита без оплаченного entitlement. Сначала нужна миграция и переходный период.

## Обязательные проверки

- Free: 5-й участник проходит, 6-й блокируется; ИИ блокируется прямым API-запросом.
- Team 50: 50-й проходит, два параллельных запроса на 51-е место не проходят.
- Business 100: аналогично для 100/101.
- Резерв приглашения учитывается и освобождается по TTL.
- Понижение ниже фактического числа мест не применяется.
- Повторный webhook идемпотентен; webhook с неверной подписью отклоняется.
- Пользователь другой организации не может читать или менять entitlement.
- Клиентская подмена `planCode`, `seatLimit` или `aiEnabled` отклоняется правилами и не влияет на сервер.
