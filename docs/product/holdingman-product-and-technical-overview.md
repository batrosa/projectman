# ProjectSfera: актуальный контекст продукта и технической реализации

Последняя проверка: 23.07.2026.

Этот файл — основной handoff для новой сессии разработки. Перед изменениями нужно
сверять его с кодом и историей Git: документ фиксирует подтверждённое состояние на
дату выше, но не заменяет проверку текущей ветки и production.

## 1. Коротко о продукте

ProjectSfera — система управления организациями, проектами, задачами, сроками,
ответственными, файлами и уведомлениями. Один пользователь может состоять в
нескольких организациях. Основные клиенты:

- web: обычные `index.html`, `style.css`, `script.js` без frontend-сборщика;
- iOS: нативное SwiftUI-приложение в каталоге `ios`;
- server API: Vercel Functions в `api` и общая серверная логика в `lib`;
- данные и realtime: Firebase Auth и Firestore;
- файлы: Cloudinary с серверной подписью;
- уведомления: лента Firestore, email, Telegram и iOS push;
- ИИ-агент: OpenRouter и данные организации/проектов.

Канонический production-домен: `https://projectman.online`.

## 4. Роли и доступ

Роли организации:

- `owner` — владелец;
- `admin` — администратор;
- `moderator` — модератор;
- `employee` — исполнитель;
- legacy `reader` встречается в старых данных и не должен давать расширенные
  полномочия.

Источник истины для роли — членство пользователя в организации и серверно
проверенный профиль. Привязка Telegram или повторный вход не должны менять:

- `organizationId`;
- `orgRole`;
- `allowedProjects`;
- существующие членства;
- подтверждённые имя и фамилию.

Владелец и администратор имеют полный доступ к обычным задачам организации.
Ограничения по проектам применяются к модераторам и исполнителям.

Инцидент с Вячеславом и Олегом ИТ оказался не сбросом роли: пользователь вошёл в
другой ProjectSfera-аккаунт и привязал Telegram к нему. Поэтому до изменения Rules
или ролей при похожей жалобе нужно сначала сравнить Firebase UID, email,
organization membership и Telegram reservation.

## 5. Авторизация и профиль

Поддерживаются:

- email и пароль;
- Google;
- Apple;
- Telegram custom token.

`lib/auth-profile.js` использует `Firebase Auth providerData` как источник истины
для native-провайдеров. Firestore-поля `authProvider` и `authProviders`
нормализуются при входе. Это исправляет старые профили, где привязка Telegram
перезаписала основной provider на `telegram`.

Если в legacy-профиле уже есть корректные имя и фамилия, он автоматически
получает `profileCompleted: true`. Существующего пользователя не должны повторно
просить заполнить имя.

Подтверждённое продуктовое правило: аккаунты разных способов входа автоматически
не объединять. Если email зарегистрирован через пароль, пользователь должен
получить понятное сообщение о входе через пароль или восстановлении пароля.

Открытый нюанс: web сейчас обрабатывает
`auth/account-exists-with-different-credential`, но Firebase может считать Google
авторитетным provider для Gmail/Google-hosted email и не всегда вернуть именно
эту ошибку. Глобальная политика «Google никогда не объединяет password-account»
ещё не гарантирована полностью. Нужна отдельная проверка provider до sign-in и
тесты всех комбинаций Google/Apple/password.

## 6. Telegram: вход, привязка и отвязка

Telegram используется как:

- самостоятельный вход;
- дополнительный способ входа;
- дополнительный канал уведомлений.

Реализовано:

- подключение Telegram в web и iOS;
- вход через Telegram-бота;
- отвязка в web и iOS;
- команда бота `/unlink`;
- запрет отвязки, если Telegram — единственный способ входа;
- сохранение native provider при привязке;
- detached reservation после отвязки, чтобы старый Telegram не создал теневой
  `tg_*` аккаунт;
- повторная привязка Telegram к нужному аккаунту после native-входа.

Общая серверная логика находится в `lib/telegram-account.js`.

Открытые нюансы:

- если в старом профиле осталось `authProviders: ["telegram"]`, но `telegramId`
  уже отсутствует, iOS может ошибочно показывать Telegram как привязанный;
- ветка `alreadyUnlinked` пока не ремонтирует такой stale Firestore-профиль;
- параллельные незавершённые link-сессии требуют дополнительной проверки на
  единственность reservation.

## 7. Удаление аккаунта

Кнопка удаления есть в личном кабинете web и iOS. Требуется двойное
подтверждение с точной фразой:

```text
УДАЛИТЬ АККАУНТ
```

Сервер требует недавнюю авторизацию: не старше 10 минут.

Перед удалением API показывает:

- количество организаций, где пользователь владелец;
- количество их проектов;
- количество участников.

Если аккаунт удаляет владелец:

- удаляются все принадлежащие ему организации;
- удаляются проекты;
- обычные и приватные задачи;
- файлы проектов и задач;
- Cloudinary-объекты;
- связанные уведомления, запросы переноса и audit-метаданные;
- членства;
- затем профиль Firestore;
- Firebase Auth удаляется последним.

Если пользователь не владелец:

- удаляются его членства;
- счётчики участников корректируются;
- из сохранённых задач удаляется его участие;
- авторство и история анонимизируются как `Удалённый пользователь`.

Физические Cloudinary-объекты удаляются раньше метаданных. Если Cloudinary
возвращает реальную ошибку, Firestore-метаданные сохраняются для безопасного
повтора.

Подтверждённые пробелы текущего cascade:

- не удаляется документ `agentRateLimits/{uid}`;
- не удаляются `agentProposalExecutions`, где UID/organization находятся в
  полях документа;
- старые Telegram login sessions могут остаться, если прежний `telegramId` уже
  был отвязан и больше не обнаруживается по профилю;
- cascade выполняется синхронно внутри `api/org.js` с `maxDuration: 60`;
  крупная организация может потребовать идемпотентный повтор.

До массового запуска удаление лучше перевести в background job с прогрессом и
полным inventory коллекций.

## 8. Обычные и приватные задачи

Обычные задачи хранятся в `tasks`.

Приватные задачи хранятся отдельно в `privateTasks`. Отдельная коллекция нужна,
потому что Firestore Rules не могут безопасно отфильтровать смешанную коллекцию
после выполнения широкого client query.

Флаг «Приватная задача» доступен при создании и редактировании в web и iOS.

Приватную задачу видят:

- владелец организации всегда;
- постановщик;
- дополнительные постановщики;
- исполнители.

Не связанный с задачей администратор, модератор или исполнитель её не видит.
Это намеренное правило. Нельзя расширять доступ администратору только потому,
что у него роль `admin`.

Для приватной задачи каноническая аудитория фиксируется в `viewerIds` и должна
совпадать с объединением:

```text
createdByUid + assigneeIds + coCreatorIds
```

Все server endpoints, уведомления, агент, монитор, перенос срока, XP, web и iOS
должны передавать `taskCollection` и учитывать обе коллекции.

Завершённая задача переходит в архив и не показывается среди активных задач.

## 10. Файлы и Cloudinary

Публичная unsigned-загрузка больше не используется для новых файлов.

Текущий поток:

1. Авторизованный клиент запрашивает server-signed upload.
2. Сервер создаёт upload intent и подписывает параметры.
3. Cloudinary хранит объект с delivery type `authenticated`.
4. Firestore хранит защищённую ссылку:
   `publicId`, `resourceType`, `deliveryType`, а не постоянный публичный URL.
5. Для открытия клиент запрашивает короткоживущий signed download после
   проверки пользователя, организации, проекта и задачи.
6. Загрузка, скачивание и удаление пишутся в audit-коллекции.

Preset:

```text
projectman_private
```

Файлы постановки задачи, completion proofs и файлы проекта используют общий
защищённый endpoint. Web и iOS умеют открывать защищённые completion proofs в
информации о задаче.

Legacy публичные Cloudinary URL распознаются отдельно для совместимости и
физического удаления. Нельзя утверждать, что все исторические объекты уже
удалены, пока это не подтверждено Cloudinary Admin API/Media Library.

Открытый P0: антивирус, карантин и content scanning отсутствуют.

## 11. Канбан, Гант и календарь

### Канбан

Активные колонки:

- «Назначенные»;
- «В работе»;
- «На проверке».

Завершённые задачи находятся в архиве. Отображение в основном канбане и модалке
«Мои задачи» должно использовать одинаковые визуальные правила статусов.

### Гант

Гант показывает только активные задачи: назначенные, в работе, на проверке и
просроченные. Завершённых задач там нет.

Цвет не зависит от статуса задачи. Он рассчитывается динамически по доле
прошедшего времени между датой создания и дедлайном:

- `early`: зелёный, «Начало срока»;
- `middle`: жёлтый, «Половина срока»;
- `late`: оранжевый, «Срок подходит»;
- `overdue`: красный, «Просрочено».

В легенде не показываются проценты. Границы расчёта в коде соответствуют
согласованной логике 0–25, 26–50, 51–99, 100+.

Переключение масштаба выполняется только кнопками «Год» и «Месяц» в toolbar.
По умолчанию выбран текущий год; для месяца — текущий месяц. Клик по названию
месяца в самой диаграмме больше не должен менять масштаб.

### Календарь

- сетка всегда семиколоночная;
- ширина дней не меняется из-за длинного названия;
- названия задач обрезаются;
- показываются только дни выбранного месяца;
- слоты до первого и после последнего дня остаются пустыми;
- пустые слоты не имеют активной рамки и немного темнее актуальных дней.

Не возвращать даты и задачи соседних месяцев в текущую сетку.

## 15. Production и инцидент Vercel

Перед production deploy, rollback, smoke, нагрузочной проверкой или изменением
Firewall обязательно прочитать:

```text
docs/vercel-protection-incident-runbook.md
```

Подтверждённые факты:

- Vercel Attack Mode во время инцидента был выключен и, по словам владельца,
  всегда был выключен;
- крупное изменение кода само не включает Attack Mode;
- burst deploy/probe/refresh может совпасть с сигналами автоматического DDoS
  mitigation, но алгоритм Vercel не раскрыт;
- `403` и `x-vercel-mitigated: challenge` — edge challenge, запрос не дошёл до
  ProjectSfera;
- `500 FUNCTION_INVOCATION_FAILED` — crash функции, это другой инцидент;
- обычный `401` JSON без токена подтверждает, что API route жив.

Браузер иногда проходит JavaScript checkpoint. iOS `URLSession`, `curl`,
webhook и Telegram bot не могут надёжно пройти его. Поэтому web может работать,
а iOS получать 403.

При `x-vercel-mitigated: challenge`:

1. прекратить повторные production-запросы;
2. не делать redeploy ради снятия challenge;
3. ждать и проверять единичным запросом с backoff;
4. не обещать точное время снятия;
5. при длительном инциденте собрать headers, `x-vercel-id`, время и обратиться в
   поддержку Vercel.

Запрещено помещать bypass secret в web/iOS и пытаться решать JS challenge из
`URLSession`.

Текущее решение владельца: оставаться на Vercel. Миграция на MyArena не
выполнялась, DNS и API base URL не менялись.

## 16. Production release-порядок

Для runtime-изменения:

1. изучить архитектуру и актуальные ограничения;
2. изменить минимальный объём кода;
3. выполнить подходящие unit/API/web/rules/iOS проверки;
4. commit и push;
5. один production deploy законченного change set;
6. один безопасный smoke по runbook;
7. Firestore Rules публиковать отдельно только при их изменении.

Не запускать быстрые циклы запросов к `projectman.online`.

Документационные изменения, которые не меняют runtime, не требуют нового
Vercel production deploy. Лишний deploy создаёт риск и не меняет сайт.

- signed authenticated Cloudinary upload/download;
- серверная проверка доступа к файлу;
- физическое удаление Cloudinary-объектов в основных cascade;
- Firestore Rules и отдельная коллекция приватных задач;
- account deletion в web/iOS;
- фиксированные SwiftPM-версии и отслеживаемый lock-файл;
- `.env.example` с актуальными переменными;
- CI для тестов, Rules, audit, build и iOS;
- XSS-регрессии для имён и URL вложений.

Что остаётся P0/P1:

- production monitoring и alerting не подключены;
- проверенные автоматические backups и restore drill отсутствуют;
- privacy/terms остаются черновиками;
- account deletion имеет перечисленные пробелы inventory и синхронный timeout;
- антивирус/карантин файлов отсутствует;
- billing и server entitlement отсутствуют;
- тарифы на landing пока декларативны;
- web Firebase SDK 9.22.0 vendored вручную;
- CSP и полный набор security headers отсутствуют в актуальном `vercel.json`;
  ограниченный CSP есть только для Firebase-hosted `google-auth.html`;
- нет полного разделения production/staging;
- смешанный брендинг ProjectSfera/HoldingMan остаётся;
- `readme.md`, `docs/product/mass-launch-readiness.md`,
  `docs/saas-readiness.md` и часть roadmap всё ещё содержат старые URL и
  устаревшие выводы;
- требуется декомпозиция `script.js` и `style.css`.

## 18. Приоритеты следующей сессии

Перед новым изменением:

1. получить `git status`, текущую ветку и `git log`;
2. проверить, включён ли нужный production commit в `main`;
3. не трогать пользовательские dirty-файлы;
4. прочитать Vercel runbook;
5. проверить, не действует ли `x-vercel-mitigated: challenge`;
6. выбрать одну законченную задачу и один deploy.

Рекомендуемый технический порядок:

1. закрыть пробелы удаления аккаунта и перевести cascade в job;
2. исправить stale Telegram provider/reservation;
3. завершить общую policy Google/password/Apple без auto-merge;
4. исправить старый URL hourly monitor;
5. подключить monitoring;
6. настроить и проверить backups/restore;
7. закончить legal-контур;
8. добавить CSP/security headers через preview;
9. обновить vendored Firebase Web SDK или перевести web на управляемые
   зависимости;
10. начать декомпозицию монолитов;
11. только после этого подключать billing и тарифные ограничения.

## 19. Нельзя забывать

- Владелец всегда видит приватные задачи; unrelated admin — нет.
- Удаление owner-аккаунта удаляет принадлежащие ему организации.
- Telegram — дополнительный provider, а не замена password/Google/Apple.
- Роль нельзя определять по визуальному способу входа.
- Для расследования аккаунта сначала сравниваются UID и membership.
- Файл считается защищённым только через server-signed flow.
- Завершённая задача не показывается в активном Ганте.
- Кнопки lifecycle в task info должны работать у исполнителя.
- Web deploy не обновляет iOS binary.
- Working browser не доказывает доступность API для iOS.
- 403 challenge не лечится повторным deploy.
- Любое утверждение о production подтверждается dashboard/logs, а не
  предположением.
